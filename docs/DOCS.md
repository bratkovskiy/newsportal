# Документация по импорту RSS

## Правило обложки (cover image)

- Импортируем только RSS-элементы `<item>`.
- Если у элемента **есть** тег `<custom:image>https://...webp</custom:image>`:
  - статья создаётся **сразу опубликованной** (`status = published`, `_status = published`);
  - ссылка из `<custom:image>` кладётся в поле `feedImageUrl` в коллекции `Articles`;
  - на сайте это изображение используется как обложка (в карточках на главной и в шапке статьи).
- Если у элемента **нет** тега `<custom:image>`:
  - статья создаётся в статусе **черновика** (`status = draft`, `_status = draft`);
  - такие статьи **не попадают на сайт**, потому что фронтенд запрашивает только `_status = published`;
  - редактор может вручную добавить обложку/контент и сменить статус на `published` в админке.

Это правило действует как для импорта из CMS (кнопка в админке), так и для фонового импорта из сервиса jobs.

## Автообновление RSS-фидов (cron + сервис jobs)

- В Payload CMS есть коллекция **Feeds**, у каждой записи (фида) есть поля:
  - `enabled` — включён ли фид для импорта;
  - `type` — тип источника (`rss` или `json`);
  - `cron` — строка расписания в формате **cron** (5 полей: минута, час, день месяца, месяц, день недели);
  - `lastImportedAt` — дата и время последнего импорта;
  - `lastImportStatus` — текстовый статус последнего импорта (создано/пропущено, ошибка и т.п.).

- Фоновый сервис **jobs** периодически опрашивает CMS и запускает импорт:
  - интервал опроса задаётся переменной окружения `FEEDS_IMPORT_INTERVAL_MINUTES` (по умолчанию `1` минута);
  - сервис выбирает только фиды с `enabled = true` и `type = rss`;
  - для каждого такого фида он смотрит поле `cron` и дату `lastImportedAt`;
  - если по cron «пора» запускать импорт (и последний импорт был раньше ожидаемого срабатывания cron) — вызывается эндпоинт `POST /api/feeds/:id/import` в CMS;
  - авторизация между jobs и CMS происходит по заголовку `x-jobs-key` с ключом из переменной окружения `JOBS_API_KEY`.

- Поведение поля `cron` в админке CMS:
  - **пустое `cron` → автообновление выключено**. Такой фид не будет импортироваться по расписанию, только вручную кнопкой в админке;
  - **заполненное `cron` → автообновление включено**. Формат стандартный Unix‑cron из 5 полей.

- Примеры cron-расписаний:
  - `* * * * *` — каждый минутный тик (каждую минуту);
  - `*/5 * * * *` — каждые 5 минут;
  - `0 * * * *` — каждый час, в начале часа;
  - `0 3 * * *` — каждый день в 03:00 по времени сервера;
  - `0 9 * * 1-5` — по будням в 09:00.

- Важные замечания:
  - если cron-строка заполнена, но **некорректна**, сервис jobs пишет ошибку в свои логи и не запускает автоимпорт для этого фида;
  - поле `lastImportedAt` обновляется и при ручном импорте (кнопка в админке), и при автоматическом импорте через jobs;
  - в админке Payload поле `lastImportedAt` отображается **с датой и временем** в формате `dd.MM.yyyy HH:mm` и доступно только для чтения;
  - поле `lastImportStatus` показывает последний результат импорта (успешно/ошибка, сколько создано и сколько пропущено);
  - UI админки не обновляется сам при автоимпорте, чтобы увидеть новое время и статус, страницу фида нужно **перезагрузить** в браузере.

## Маппинг полей и настройка фидов

Для каждого фида можно настроить соответствие полей фида полям статей (маппинг):

- **Поля источника:** заголовок, контент, анонс, автор, категория, теги, изображение, дата
- **Дефолтная категория:** если категория не найдена в фиде, используется указанная категория (по умолчанию ID 64)
- **Источник автора:** выбор из трех вариантов:
  - **Из фида** - автор берется из указанного поля (например, `dc:creator`), создается новый или используется существующий по имени
  - **Существующий автор** - выбор из списка авторов в базе, все статьи получат этого автора
  - **Создать нового автора** - ввод имени, автор создается при первом импорте и назначается всем статьям

Подробнее см. `FEED_AUTHOR_MAPPING.md`

## Страницы категорий

Каждая категория имеет собственную страницу с URL: `/categories/{slug}`

### Автогенерация полей:
- **Slug (URL алиас)** - автоматически из названия категории (транслитерация, lowercase, дефисы)
- **Meta Title** - шаблон: `{НАЗВАНИЕ КАТЕГОРИИ} - LADY.NEWS`
- **Meta Description** - шаблон: `Все материалы из категории {НАЗВАНИЕ КАТЕГОРИИ}`

Все автосгенерированные поля можно редактировать вручную в админке.

### Меню навигации:
- В шапке сайта динамически загружаются категории из CMS
- Показываются первые 10 категорий по алфавиту
- Каждая ссылка ведет на страницу категории

### Функционал страницы:
- Показываются только статьи выбранной категории
- Статус `published`
- Сортировка по дате публикации (новые первыми)
- Аналогичный главной странице дизайн

Подробнее см. `CATEGORIES_PAGES.md`

## Счетчики просмотров статей

Каждая статья имеет счетчик просмотров, который автоматически увеличивается при каждом просмотре.

### Функционал:
- **Автоматический подсчет** - инкремент при загрузке страницы статьи
- **Атомарность** - корректная работа при одновременных просмотрах
- **Отображение в CMS** - поле "Просмотры" в сайдбаре статьи (только чтение)
- **Автоудаление** - счетчик удаляется вместе со статьей

### API:
- `POST /api/articles/:id/increment-view` - инкремент счетчика
- Сортировка: `GET /api/articles?sort=-viewCount` - самые популярные

### База данных:
- Колонка `view_count` (integer, default 0)
- Индекс для быстрой сортировки по популярности

Подробнее см. `VIEW_COUNTER.md`

## Заглушка для битых/отсутствующих изображений

**Технические детали:**
- Файл `placeholder-1.jpg` хранится в Docker volume `cms_media` (`/app/src/media/placeholder-1.jpg`)
- Endpoint `/api/media/file/placeholder-1.jpg` отдает файл **напрямую из файловой системы** с fallback
- **Не требует загрузки через админку** - работает автоматически при наличии файла в volume
- Файл сохраняется между перезапусками контейнера

**Использование на фронтенде:**
- На фронтенде используется единый placeholder-ресурс `placeholder-1.jpg` из CMS (`/api/media/file/placeholder-1.jpg`)
- Для карточек статей на главной странице:
  - если у статьи нет обложки (`coverImage`, `feedImageUrl`), на фронтенде подставляется URL заглушки
  - если загруженная картинка не открывается в браузере (ошибка загрузки), на клиенте срабатывает `onerror` и изображение заменяется на заглушку
- Для страницы статьи:
  - обложка статьи формируется по тем же правилам (RSS `custom:image` → `feedImageUrl` → `coverImage` → заглушка)
  - при ошибке загрузки обложки также используется заглушка
  - для `<img>` внутри HTML-тела статьи при рендеринге автоматически добавляются `loading="lazy"`, `decoding="async"` и обработчик ошибки загрузки с подстановкой заглушки
- Для всех новых страниц и компонентов, которые показывают изображения материалов, необходимо:
  - прокидывать URL заглушки `PLACEHOLDER_IMAGE_URL` (формируется на фронтенде как `new URL('/api/media/file/placeholder-1.jpg', CMS_URL).toString()`)
  - добавлять обработчик `onerror` для `<img>` с подменой на URL заглушки

## Автоподгрузка статей на главной странице

- На главной странице (`/`) используется постраничная загрузка статей с шагом **100 штук** (константа `PAGE_LIMIT` в `index.astro`).
- При первой загрузке страницы:
  - Сервер запрашивает у CMS первую страницу (`page=1`, `limit=100`).
  - На странице доступны до 100 последних опубликованных статей, из которых формируются блоки и лента.
- Дальнейшая подгрузка происходит через:
  - **Кнопку "Показать ещё"**: при клике загружается следующая порция из 100 статей.
- Для подгрузки используется API-эндпоинт `/api/frontend/articles?page=N&limit=PAGE_LIMIT`, который проксирует запросы в Payload CMS.
- Когда все статьи загружены (из ответа видно `hasNextPage: false` или `docs` пустой):
  - Кнопка "Показать ещё" скрывается.
- Новые карточки статей создаются динамически на клиенте с той же разметкой и классами, что и `CardFeed.astro`, включая обработчик `onerror` для заглушки изображений.

## Блоки на главной странице

На главной странице используется единый список до **100** последних опубликованных статей из CMS. На его основе формируются тематические блоки в строгом порядке приоритета.

### Источник данных

- CMS: коллекция `articles`.
- Выборка:
  - только статьи со статусом `published`;
  - сортировка по дате публикации (новые первыми);
  - лимит `PAGE_LIMIT = 100`.

### Приоритет блоков

1. **Главный блок** (большой квадрат слева).
2. **Блок "Главное"** (правый столбец, до 4 материалов).
3. **Блок Must-Read Stories** (до 2 материалов).
4. **Блок "Популярное"** (4 материала по просмотрам).
5. **Блоки по категориям** (до 2 категорий, каждая до 4 материалов).
6. **Лента "Последние новости"** (оставшиеся материалы без дублей с блоками выше).

Для контроля дублей используется множество `usedIds` (внутренне) и дополнительный набор `featuredIds` для исключения материалов из ленты.

### Флаги на статьях (CMS → коллекция `articles`)

У каждой статьи есть три чекбокса:

- `homeMainBlock` — отправить в **Главный блок** (большой квадрат).
- `homeTop` — отправить в блок **"Главное"**.
- `homeMustRead` — отправить в блок **Must-Read Stories**.

Правило: **у статьи может быть установлен только один из этих флагов**. Валидация Payload в `beforeValidate` не даёт сохранить материал с несколькими флагами одновременно.

### Флаг на категориях (CMS → коллекция `categories`)

- `showOnHome` — чекбокс "Показывать категорию в блоках на главной странице".
- Блок по категории появляется только если:
  - у категории `showOnHome = true`;
  - среди последних 100 опубликованных статей есть хотя бы одна с этой категорией.

Максимум отображается **2 категории** (первые по алфавиту, т.к. запрос категорий сортируется по `name`).

### Логика выбора по блокам

#### 1. Главный блок

- Если есть статьи с `homeMainBlock = true`:
  - выбирается одна статья по правилу "дата ↓, затем просмотры ↓".
- Если флаг не проставлен ни у одной статьи:
  - берётся просто самая свежая статья по тому же правилу.
- Выбранная статья добавляется в `usedIds`.

#### 2. Блок "Главное" (до 4 материалов)

Цель — показать до 4 материалов с учётом флага `homeTop` и уже использованных статей.

1. Берутся все статьи с `homeTop = true`, сортируются "дата ↓, затем просмотры ↓".
2. Сначала выбираются статьи с этим флагом, которые **ещё не были использованы** (`id ∉ usedIds`).
3. Если таких меньше 4:
   - добираются остальные статьи с `homeTop = true`, даже если они уже попали в другие блоки.
4. Если всё ещё меньше 4:
   - берутся самые свежие статьи без флагов (по тому же правилу);
   - сначала те, которые не использовались, потом (при нехватке) — с повторами.

Все выбранные id добавляются в `usedIds`.

#### 3. Must-Read Stories (до 2 материалов)

Аналогично блоку "Главное", но используется флаг `homeMustRead` и лимит 2 материала:

1. Статьи с `homeMustRead = true` сортируются "дата ↓, просмотры ↓".
2. Сначала берутся неиспользованные статьи, затем при необходимости — уже использованные.
3. Если флагов нет или их мало, недостающие материалы добираются из общего списка по "дата ↓, просмотры ↓" (сначала новые, затем при необходимости с повторами).

Выбранные статьи добавляются в `usedIds`.

#### 4. "Популярное" (4 материала)

- Все статьи сортируются по "просмотры ↓, затем дата ↓".
- Сначала выбираются материалы, которые ещё не использованы (`id ∉ usedIds`), пока не наберётся 4.
- Если не хватает:
  - добираются статьи из того же списка, даже если они уже использовались выше (без дублей **внутри** блока "Популярное").

Выбранные статьи добавляются в `usedIds`.

#### 5. Блоки по категориям (до 2 категорий, каждая до 4 материалов)

Для каждой категории с `showOnHome = true` (в алфавитном порядке), пока не набрано 2 категории:

1. Находятся все статьи этой категории среди последних 100.
2. Статьи сортируются по "просмотры ↓, затем дата ↓".
3. Формируется список до 4 материалов:
   - сначала берутся статьи, которых ещё нет в `usedIds`;
   - если не хватает, добираются статьи этой же категории, даже если они уже использовались в других блоках (но без дублей **внутри** блока категории).
4. Если в результате у категории есть хотя бы одна статья, создаётся блок с заголовком = имя категории.

#### 6. Лента "Последние новости"

- Базовый список ленты — те же последние 100 статей.
- Из него **исключаются** все статьи, которые попали в:
  - Главный блок,
  - блок "Главное",
  - Must-Read Stories,
  - "Популярное",
  - блоки по категориям.
- В результате в ленте **нет дублей** с верхними блоками, но порядок ("новые первыми") сохраняется.
- Лента используется и как стартовый набор для последующей подгрузки через кнопку "Показать ещё".

## Блоки на страницах статей

Для страниц отдельных материалов используется свой набор блоков, логика которых согласована с главной страницей, но всегда исключает **текущую статью**.

### Источник данных

- Для блоков на странице статьи запрашиваются до 100 последних опубликованных материалов из CMS (`status = published`, сортировка по дате публикации, лимит 100).
- Из выборки всегда удаляется статья с текущим `id`.

### Блок "Главное" (в правой колонке)

- Отображается в правой колонке (сайдбар) **над рекламным блоком**.
- Использует те же флаги, что и главная:
  - учитывается флаг `homeTop` на статьях;
  - приоритет отдается материалам с этим флагом.
- Правила выбора (до 4 материалов):
  1. Берутся статьи с `homeTop = true`, сортировка "дата ↓, затем просмотры ↓".
  2. Сначала выбираются статьи, которые ещё не были использованы на странице (уникальный `id`).
  3. Если таких меньше 4 — добираются остальные статьи с `homeTop = true`.
  4. Если всё ещё меньше 4 — добираются самые свежие статьи без флага по тому же правилу.
- Текущая статья никогда не попадает в этот блок.

### Блок "Читайте также" (под статьёй)

- Находится под основным текстом статьи в основной колонке.
- Показывает до **6** материалов.
- Источник: те же последние 100 статей **за вычетом** текущей статьи и материалов, уже попавших в блок "Главное" на этой странице.
- Сортировка: только по дате публикации (новые первыми), аналогично ленте "Последние новости" на главной.
- Вёрстка повторяет карточки из ленты на главной (2 колонки на десктопе, изображение + категория/тег + заголовок + автор).

### Блок "Ещё по теме" (под "Читайте также")

- Находится под блоком "Читайте также".
- Формируется по **первому тегу** текущей статьи:
  1. Находим тег по `slug` и его `id`.
  2. Запрашиваем статьи, где этот тег присутствует (`where[tags][equals]=tagId`) и статус `published`.
  3. Исключаем текущую статью по `id`.
  4. Берём до 6 материалов, сортируя по дате публикации (новые первыми).
- Вёрстка блока такая же, как до внедрения новых блоков: сетка из 2 колонок на десктопе, с картинкой, тегом/категорией, заголовком и автором.

## Хранение медиафайлов

Медиафайлы CMS сохраняются в Docker volume `cms_media`, который монтируется в `/app/src/media` внутри контейнера.

**Важно:**
- Volume сохраняет файлы между перезапусками контейнера
- `placeholder-1.jpg` уже находится в volume и **работает автоматически** без загрузки через админку
- Для загрузки других медиафайлов (обложки статей) используйте админку CMS
- Для бэкапа и восстановления медиа см. `MEDIA_SETUP.md`

**Структура:**
```
cms_media (Docker volume)
├── placeholder-1.jpg        # Файл-заглушка (автоматический fallback)
├── <filename>-<hash>.jpg    # Загруженные изображения
├── <filename>-<size>.jpg    # Ресайзы (thumbnail, card, tablet)
```

**Восстановление placeholder при необходимости:**
```bash
# Скопировать файл в volume
docker cp placeholder-1.jpg news_project-cms-1:/app/src/media/placeholder-1.jpg

# Или использовать скрипт
./RESTORE_PLACEHOLDER.sh
```

## Настройки robots.txt через CMS

Файл `robots.txt` теперь управляется через CMS и отдаётся фронтендом динамически.

### Где править robots.txt

- В админке CMS откройте раздел:
  - **Настройки → robots.txt** (global `robots`).

В этом разделе одно поле:

- **`content`** — многострочное поле (textarea) с полным текстом файла `robots.txt`.

### Как это работает на фронтенде

- Фронтенд-роут `/robots.txt` реализован в файле:
  - `/apps/web/src/pages/robots.txt.ts`.
- При запросе браузера к `/robots.txt` происходит следующее:
  1. Фронтенд запрашивает `http://cms:3000/api/globals/robots` (в dev — `http://localhost:3000/...`).
  2. Если глобал доступен и `content` не пустой, отдаётся именно это содержимое.
  3. Если глобал пустой или недоступен (ошибка сети/бэкенда), используется дефолтный вариант:

```text
User-agent: *
Allow: /

Sitemap: {SITE}/sitemap.xml
```

где `{SITE}` берётся из переменной окружения `SITE` (см. `.env` и `DEPLOY.md`).

Таким образом, для изменения `robots.txt` достаточно отредактировать поле `content` в админке и сохранить изменения — фронтенд начнёт отдавать новый текст без переката кода.

## Мультиязычность (RU / EN)

В проекте реализован простой глобальный переключатель языка сайта (RU/EN) через CMS. Переводятся только общие элементы интерфейса — заголовки блоков и текст футера, а также формат даты на страницах статей.

### Где настраивается язык

- В админке CMS откройте раздел:
  - **Настройки → Настройки сайта / язык** (global `site-settings`).

Поля:

- **`currentLanguage`** — текущий язык сайта:
  - `ru` — русский интерфейс (значение по умолчанию);
  - `en` — английский интерфейс.

Группа **`labels`** в `site-settings` управляет заголовками блоков:

- `homeMainRu` / `homeMainEn` — заголовок блока **«Главное»** на главной и в сайдбаре статьи.
- `homeLatestRu` / `homeLatestEn` — заголовок блока **«Последние новости»** на главной.
- `homeMustReadRu` / `homeMustReadEn` — заголовок блока **Must-Read Stories** на главной.

При смене `currentLanguage` фронтенд автоматически выбирает RU или EN варианты вышеуказанных полей.

На страницах материалов используется компонент `ArticleLayout.astro`, который форматирует дату в зависимости от языка:

- при `currentLanguage = 'en'` — формат `en-US` (например, `December 12, 2025`).

В группе **`footer`** глобала `site-settings` задаются тексты футера (без года):

На фронтенде в футере отображается строка вида:

Язык выбирается по полю `currentLanguage` глобала `site-settings`.

## Система рекламных мест и режим ?adv

### Где настраиваются рекламные места

- В админке CMS есть отдельная коллекция **Ad slots** (`ad-slots`).
- Каждый документ в `ad-slots` описывает одно рекламное место (слот) на сайте.
- Ключевые поля:
  - **`key`** — машинное имя слота (обязательно, уникально), например:
    - `sidebar_default` — правая колонка по умолчанию (главная, категории, теги, поиск, /legal и др. страницы с макетом `MainColumns`).
    - `article_sidebar` — правая колонка на странице статьи.
  - **`label`** — человекочитаемое название слота для редакторов.
  - **`description`** — описание, где именно на сайте находится слот.
  - **`enabled`** — включен ли показ рекламы в этом месте.
  - **`provider`** — рекламная сеть / тип слота:
    - `yan` — Яндекс РСЯ;
    - `adsense` — Google AdSense;
    - `direct` — прямой код (резерв);
    - `prebid` — слот под header bidding.
  - **`yandexBlockId`** — ID блока РСЯ (RTB `blockId`), используется при `provider = yan`.
  - **`adsenseSlotId`** — значение `data-ad-slot` из кода AdSense, используется при `provider = adsense`.
  - **`adFormat`** — формат AdSense (например, `auto`).
  - **`width` / `height`** — рекомендуемый размер слота в пикселях (используется для высоты плейсхолдера).

### Как включить/выключить рекламу в слоте

1. Откройте в админке раздел **Ad slots**.
2. Найдите нужный слот по полю **`key`**:
   - `sidebar_default` — правая колонка на большинстве страниц.
   - `article_sidebar` — правая колонка на странице статьи.
3. Выберите **`provider`** (РСЯ или AdSense):
   - Для РСЯ заполните **`yandexBlockId`** (ID блока RTB).
   - Для AdSense заполните **`adsenseSlotId`** (значение `data-ad-slot` из кода рекламы).
4. При необходимости укажите `width`/`height` (рекомендованный размер слота).
5. Поставьте галочку **`enabled`**, чтобы слот начал реально показывать рекламу.
6. Сохраните документ.

**Важно:**

- На фронте слот считается "рабочим", только если одновременно:
  - `enabled = true` **и**
  - корректно заполнены поля для выбранного `provider` (есть `yandexBlockId` или `adsenseSlotId`).
- Если конфигурация некорректна или слот выключен, блок полностью схлопывается и не занимает место в вёрстке.

### Режим подсветки рекламных мест (?adv)

Для отладки размещения рекламы есть специальный режим подсветки слотов.

- Чтобы включить режим, откройте любую страницу сайта с параметром **`?adv`** в URL:
  - Примеры:
    - `https://lady.news/` → `https://lady.news/?adv`
    - `https://lady.news/article/some-slug-123.html` → `https://lady.news/article/some-slug-123.html?adv`
    - `http://localhost:4321/?adv` — локальный dev.
- В этом режиме:
  - **все рекламные слоты** становятся видимыми, даже если они выключены или не сконфигурированы;
  - каждый слот получает цветную рамку и фон:
    - жёлтый — слот включён и корректно настроен (есть код);
    - красный — нет кода или слот выключен;
  - внутри слота выводится служебная информация:
    - `key` — машинное имя слота;
    - `provider` — провайдер (yan / adsense / ...);
    - `status` — текстовое описание состояния (включен, выключен, нет кода).
- В режиме `?adv` реальные рекламные сети **не вызываются**, баннеры **не загружаются** — показывается только отладочная разметка.

Это удобно для верстальщиков и редакторов: видно все потенциальные рекламные места, даже если рекламы ещё нет.

### Для разработчиков: новые шаблоны с рекламой

Если добавляется **новый тип страницы или layout** и там нужно вывести рекламный слот:

- В большинстве существующих layout’ов (`MainColumns.astro`, `Article.astro` и т.п.) компонент `AdSlot` **уже импортирован**.
- В таких шаблонах достаточно вставить строку:

  ```astro
  <AdSlot placementKey="sidebar_default" />
  ```

  или другой ключ (`article_sidebar`, `home_inline_1` и т.п.) — он должен совпадать с полем `key` слота в CMS.

- Если создаётся **совсем новый .astro-файл**, где `AdSlot` ещё не использовался, в верхнем блоке между `---` нужно один раз добавить импорт:

  ```astro
  ---
  import AdSlot from '../components/AdSlot.astro'; // путь может отличаться
  // другие импорты и логика
  ---
  ```

  После этого в разметке можно использовать любые слоты через `<AdSlot placementKey="..." />`.

## Произвольный HTML-код в <head> через админку

Для вставки дополнительного кода в `<head>` (пиксели, мета-теги, внешние скрипты) используется глобал **AnalyticsSettings**.

- В админке откройте **Globals → Аналитика и веб-мастеры (analytics-settings)**.
- Внизу формы есть поле:
  - **`customHeadHtml`** — «Произвольный HTML-код в `<head>`».
- Любой HTML, который вы вставите в это поле, будет **добавлен как есть** внутрь тега `<head>` на **всех страницах сайта**.

Типичные сценарии использования:

- Код пикселей/рекламных систем, которые не покрыты стандартными настройками Метрики/GA.
- Дополнительные `<meta>`-теги валидации/интеграций.
- Настройки A/B‑тестирования, виджеты и т.п.

**Ограничения и безопасность:**

- Не вставляйте сюда теги `<html>`, `<head>`, `<body>` — только их содержимое.
- Код не фильтруется и не экранируется, поэтому вставляйте **только доверенный HTML/JS**.
- Изменение этого поля не требует пересборки фронта — после сохранения глобала эффект виден сразу.

## Текстовый логотип и подзаголовок в шапке

Настройка текстового логотипа и второй строки (подзаголовка) в шапке задаётся через глобал **Site settings / язык** (`site-settings`).

В админке: **Globals → Настройки сайта / язык**.

- **`siteName`** — название сайта, используется как **текстовый логотип** в шапке и слева в футере.  
  - Можно использовать простой текст: `MODA.LA`.  
  - Можно вставлять HTML, например: `MODA<span class="logo-accent">.LA</span>` — этот HTML рендерится только в шапке.
- **`headerTagline`** — *необязательная* вторая строка под логотипом (подзаголовок), например: `NEWS ABOUT MODA`.
- **`headerLogoCss`** — произвольный CSS, который попадает в `<head>` внутри `<style>` и может управлять стилем логотипа.

Как это используется на фронте:

- В шапке:
  - основная строка (`siteName`) рендерится как HTML внутри элемента с классом `.site-logo-header-main`;
  - если заполнен `headerTagline`, он выводится второй строкой в `.site-logo-header-subline`;
  - общий контейнер имеет класс `.site-logo-header`.
- В футере и в копирайте используется **очищенный текст** названия (без HTML‑тегов).

Пример CSS, который можно вставить в `headerLogoCss`:

```css
.site-logo-header {
  display: inline-flex;
  flex-direction: column;
  align-items: flex-start;
  line-height: 1.1;
}

.site-logo-header-main {
  font-family: 'Playfair Display', serif;
  font-weight: 700;
  font-size: 24px;
  letter-spacing: 0.25em;
  text-transform: uppercase;
}

.site-logo-header-subline {
  margin-top: 4px;
  font-family: 'Inter', system-ui, sans-serif;
  font-size: 11px;
  letter-spacing: 0.18em;
  text-transform: uppercase;
  color: #71717a;
}

.site-logo-header .logo-accent {
  color: #f97316;
}
